# 迅虎支付模式观察

## 它怎么做

迅虎支付/虎皮椒公开文档展示的是第三方支付网关模型，而不是把个人支付宝静态收款码改造成官方回调。

典型链路是：

1. 商户在迅虎平台开户，获得 `APPID` 和 `SECRET`。
2. 商户服务端调用迅虎下单接口，提交订单号、金额、商品标题、`notify_url`、随机串和签名。
3. 付款人跳转或扫码完成支付。
4. 迅虎平台收到支付结果后，由迅虎服务器 POST 商户传入的 `notify_url`。
5. 商户校验迅虎回调签名、金额和订单状态，再发货或开通权益。

公开 API 文档里，下单接口示例为 `https://api.xunhupay.com/payment/do.html`，支付成功后会向商户 `notify_url` 发起 POST 通知，并说明失败会重试。

## 可借鉴点

- 平台侧分配 `APPID/SECRET`，调用接口必须签名。
- 商户下单必须传业务订单号和异步通知地址。
- 回调 payload 要带交易号、订单号、金额、状态和签名。
- 商户必须做金额核对、订单幂等和签名校验。
- 回调失败需要重试和投递记录。

## 本项目对应设计

本项目借鉴的是“网关 + 商户 API + 签名 Webhook”的结构：

- `MERCHANT_TOKEN` 保护订单创建和查询。
- `ADMIN_TOKEN` 保护人工确认、取消和审计查询。
- `WEBHOOK_SECRET` 签名发给商户的 Webhook。
- `webhookDeliveries` 记录投递状态。
- 文档明确个人码只做人工确认，不声称官方回调。

## 不能照搬的点

迅虎是第三方支付平台，是否可用取决于它自身资质、签约范围、支付通道和风控规则。普通开发者不能只靠个人收款码复刻它的“平台回调”能力；如果没有官方支付通道或合规第三方通道，个人码场景仍只能做人工核验和账务记录。
